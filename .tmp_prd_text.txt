1. Executive Summary 
Paritium is a web-based exchange rate comparison platform designed to help consumers and 
businesses identify the most competitive foreign exchange (FX) rates across multiple currency 
exchange providers. The platform aggregates live rates from integrated exchange platforms, 
enabling users to make informed, transparent decisions about their currency conversions and 
international money transfers. 
 
This PRD defines the functional and non-functional requirements for the development of the 
Paritium website. It captures the six core development tasks derived from stakeholder input and

2. Product Overview 
2.1 Product Vision 
To be the most trusted, transparent, and user-friendly platform for comparing real-time foreign 
exchange rates across multiple providers — empowering users to maximise the value of every 
currency transaction. 
 
2.2 Product Goals 
• Aggregate live exchange rates from multiple FX platforms in one unified interface 
• Enable users to compare rates side-by-side and identify the best deal 
• Display survey links and app download links for partner transfer providers 
• Capture user feedback and behaviour data to continuously improve the product 
• Establish a scalable, secure technical foundation that supports future growth 
• Provide a seamless bridge between the website and the Paritium mobile app upon 
launch 
 
2.3 Target Users 
• Individuals sending money internationally (remittance users) 
• Travellers seeking competitive rates before currency exchange 
• Small businesses managing multi-currency payments 
• Diaspora communities monitoring home-currency rates (including NGN and other 
emerging market currencies)

4. Project Scope 
4.1 In Scope 
• Website development at paritium.com 
• Live exchange rate API integrations (multiple providers) 
• Daily/periodic rate update mechanism 
• Survey links to partner transfer provider surveys, embedded on the website

6. Functional Requirements 
The following six tasks represent the core functional requirements for the Paritium website, as 
defined by the product stakeholders at Greenvine Technologies and EccentricLLC. 
 
T1 Live Exchange Rate API Integration 
 
Overview 
The website must integrate with the public APIs or rate-scraping endpoints of exchange 
platforms (e.g., Wise, Remitly, WorldRemit, TapTap, Lemfi, and others) to fetch live FX rates 
and display them in real time to website visitors. 
 
Functional Requirements 
• Integrate with a minimum of 3 exchange platform APIs at launch, with capacity for more 
• Display live exchange rates per provider on the homepage rate comparison table 
• Implement a scheduled rate-refresh mechanism (minimum: every 60 minutes; target: 
real-time or near-real-time) 
• Show timestamp of last rate update for each provider 
• Handle API failures gracefully — display last known rate with a staleness indicator 
• Support configuration of which currency pairs to display (e.g., USD/NGN, GBP/NGN, 
EUR/NGN, CAD/NGN) 
 
Acceptance Criteria 
◦ Rates from at least 3 providers are displayed on page load without manual refresh  
◦ Rate update timestamps are accurate and reflect the last successful API call 
◦ If an API is down, the page does not crash — a fallback message is shown 
◦ Rate refresh can be triggered manually by the admin without a full deployment 
 
T2 Survey Integration & Lead Capture 
 
Overview 
The website must embed or link to survey tools to collect user feedback and capture data about 
provider experiences. This includes a Paritium platform survey and individual provider-specific 
survey links. Survey responses must be stored and accessible for analysis by the Greenvine 
Technologies and EccentricLLC product team. 
 
Functional Requirements 
• Embed or display a prominent Paritium survey link/widget (homepage, /survey page, 
post-rate-check screen)

T3 Web Analytics & User Behaviour Tracking 
 
Overview 
Web analytics must be implemented to give Greenvine Technologies and EccentricLLC full 
visibility into how many users visit the website, which providers attract the most interest, which 
app download links are clicked, and how users navigate the site. This data will drive product 
decisions, provider prioritisation, and marketing strategy. 
 
Functional Requirements 
• Implement Google Analytics 4 (GA4) as the primary analytics platform 
• Supplement GA4 with a secondary tool such as Hotjar or Microsoft Clarity for session 
recordings and heatmaps 
• Track the following USER VISIT metrics: 
◦ Total daily, weekly, and monthly unique visitors 
◦ New vs returning visitors 
◦ Session duration and pages per session 
◦ Bounce rate per page 
◦ Traffic sources: organic search, direct, referral, social media, paid 
◦ Geographic location of visitors (country, city) 
◦ Device type: mobile, tablet, desktop 
◦ Browser and operating system 
• Track the following USER TRANSACTION / INTERACTION events: 
◦ Currency pair selected (e.g. user selected USD → NGN) — log the pair chosen 
◦ Provider row viewed or expanded — log which provider and which currency pair

T4 Scalable API Architecture 
 
Overview 
The website's backend must be architected to support the seamless addition of new exchange 
platform API integrations over time, without requiring major refactoring or downtime. 
 
Functional Requirements 
• Design a modular API connector layer where each exchange provider is a self-contained 
integration module 
• Use a standardised data schema for all rate data (provider name, currency pair, buy 
rate, sell rate, timestamp, source URL, app store links) 
• New providers must be addable by updating a configuration file or adding a new 
connector module — no changes to core frontend logic

T5 Secure Infrastructure 
 
Overview 
The platform must be built and hosted on a secure, reliable infrastructure that protects user 
data, prevents unauthorised access, and meets industry-standard security practices. 
 
Functional Requirements 
• All traffic must be served over HTTPS (SSL/TLS certificate required — e.g., Let's 
Encrypt or CDN-provided) 
• API keys and credentials for third-party services must be stored as environment 
variables — never hardcoded in the codebase 
• Implement rate limiting on any backend API endpoints to prevent abuse 
• Apply Content Security Policy (CSP), X-Frame-Options, and other standard HTTP 
security headers 
• No personally identifiable information (PII) collected or stored without explicit user 
consent 
• Hosting environment must support automated backups (daily minimum) 
• Regular dependency vulnerability scans (e.g., npm audit, Snyk) as part of the CI/CD 
pipeline 
• DDoS protection via CDN (e.g., Cloudflare) 
 
Acceptance Criteria 
◦ SSL certificate is active and auto-renewing at launch 
◦ Security headers are validated via securityheaders.com — minimum Grade B 
◦ No API keys appear in the public codebase or browser network requests 
◦ Penetration testing or security audit completed prior to public launch

T6 App Download Links — Paritium & Partner Providers 
 
Overview 
The website must display App Store and Google Play download links for international transfer 
companies (e.g., Wise, Remitly, WorldRemit, TapTap, Lemfi and others) within the rate 
comparison table. Additionally, the website must reserve space for the Paritium mobile app 
download links, which will go live once the app is released. 
 
Functional Requirements — Partner Provider App Links 
• Display Apple App Store and Google Play Store icons next to each provider in the 
comparison table 
• Each icon links directly to the provider's app listing on the respective store (opens in a 
new tab) 
• App links must be stored in the provider configuration file so they can be updated without 
code changes 
• Track all provider app download link click events via the analytics layer (Task 3) 
• Display tooltip on hover: e.g. 'Download the Wise app on the App Store' 
 
Functional Requirements — Paritium App Links (Future) 
• Reserve a prominent section on the homepage and in the nav for Paritium app download 
CTAs 
• Display Apple App Store and Google Play Store badges — initially in a 'Coming Soon' 
disabled state 
• On app launch, update links to live store URLs by changing a single configuration 
variable 
• Optionally include a 'Notify me at launch' email capture for interested users 
 
Acceptance Criteria 
◦ App download icons are visible for each provider in the comparison table at launch  
◦ Each icon correctly links to the provider's own app store page 
◦ Paritium 'Coming soon' app section renders correctly on mobile, tablet, and desktop 
◦ Store links can be activated for Paritium by updating a single config variable 
◦ Click events are tracked in GA4 for both provider and Paritium app download links
